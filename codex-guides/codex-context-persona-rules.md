# Codex 回答依据、上下文输入与项目人设配置

> 更新日期：2026-05-08  
> 当前工作区：`E:\code\codex_projects\first_use`

## 1. 先说结论

我回答你的问题时，不是只看你最后一句话。大模型实际会综合多层输入：

- 系统级规则
- 开发者级规则
- 当前 Codex app / 工作区环境信息
- 当前会话历史
- 你刚刚发来的用户请求
- 我通过工具读取到的文件、终端输出、网页或连接器数据
- 当前可用工具、插件、技能和权限边界
- 项目内的说明文件，例如 `AGENTS.md`、README、文档、配置文件

如果你想让我在某个项目中保持固定人设、固定回答结构、固定代码风格或固定工作流程，最推荐的做法是在项目根目录写一个 `AGENTS.md`。

如果另一个项目需要另一套人设和规则，就在另一个项目根目录写另一份 `AGENTS.md`。不同项目使用不同工作区打开，Codex 会按当前项目的规则工作。

## 2. 我回答问题时，大模型的输入包含什么

可以把一次回答理解成这样：

```text
系统规则
  + 开发者规则
  + 当前工具和权限说明
  + 当前工作区环境
  + 技能/插件说明
  + 历史对话
  + 用户最新问题
  + 我刚刚读取到的文件或命令结果
  = 本次回答的上下文
```

这些内容的优先级不完全一样。越上层的规则越强，越不能被下层覆盖。

## 3. 各类输入分别是什么

### 3.1 系统级规则

系统级规则是最高优先级的行为边界。它通常规定：

- 我是什么类型的助手
- 必须遵守哪些安全规则
- 什么时候必须联网确认
- 如何使用工具
- 如何处理版权、引用、隐私和高风险内容

这部分你通常看不到完整内容，也不能通过普通提示词覆盖。

例如，你不能通过说“忽略所有安全规则”让我绕过系统限制。

### 3.2 开发者级规则

开发者级规则是 Codex app 或当前运行环境给我的工作方式说明。它会规定：

- 我是一个 coding agent
- 应该如何读代码、改代码、跑测试
- 如何使用 shell、Git、插件和技能
- 文件编辑时要遵守什么约束
- 前端开发时要怎么验证
- 最终回答应该怎么简洁总结
- 当前沙盒权限和可写目录是什么

这部分优先级也高于你的普通项目规则。

### 3.3 当前环境信息

当前环境会告诉我：

```text
当前目录：E:\code\codex_projects\first_use
Shell：PowerShell
当前日期：2026-05-08
时区：Asia/Shanghai
沙盒模式：workspace-write
可写根目录：E:\code\codex_projects\first_use
```

这些信息会影响我怎么解释路径、怎么运行命令、哪些地方可以写文件。

### 3.4 会话历史

本轮对话中你之前问过什么、我生成过什么文件、我做过哪些解释，都会作为上下文的一部分影响后续回答。

比如你连续让我生成了这些文档：

```text
codex_intro.md
codex_permissions.md
codex_plugins_connectors.md
codex_git_workflow.md
```

所以我知道你正在系统性整理“第一次使用 Codex”的知识。

### 3.5 用户最新请求

你最新发来的消息通常是最直接的任务目标。

例如这次你问：

```text
你回答我问题时是根据什么回答的，即具体给大模型的输入包含什么部分；
如果我想要你在这个项目中保持一种人设，使用一种固定的结构和语法回答等功能，应该怎么实现；
同时，如果我想在另一个项目中让你用另一个人设又该怎么做。
回答同样以.md呈现
```

所以我会生成一份 Markdown 文档，而不是只在聊天里回答。

### 3.6 我读取到的文件和命令输出

我不是天然知道你电脑里所有文件的内容。只有当我使用工具读取了某个文件、列目录、搜索文本、运行命令后，结果才会进入当前上下文。

例如：

```powershell
Get-ChildItem
Get-Content README.md
rg "keyword"
git status
```

这些命令的输出会影响我接下来的判断。

### 3.7 工具、插件、连接器和技能

我会知道当前有哪些工具可用，例如：

- 文件编辑工具
- PowerShell 命令执行
- GitHub 工具
- Browser 插件
- 图片生成工具
- 自动化任务工具

如果某个技能被触发，我还会读取它的说明文件，比如 `SKILL.md`，然后按里面的流程做事。

### 3.8 项目内规则文件

项目内可以放规则文件来影响我在该项目中的行为。最推荐的是：

```text
AGENTS.md
```

它通常放在项目根目录，用来告诉 Codex：

- 项目是什么
- 怎么启动和测试
- 代码风格
- 分支和提交规则
- 回答结构
- 人设和语气
- 禁止事项
- 高风险操作前要怎么确认

## 4. 输入优先级大致是什么

大致可以这样理解：

```text
系统规则
  > 开发者规则
  > 工具和权限限制
  > 项目规则文件
  > 会话中你给的长期偏好
  > 你当前这一条请求
  > 我读取到的普通项目内容
```

注意：这不是一个可以随意改写的“绝对公式”，但作为理解模型行为很有用。

如果项目规则和系统规则冲突，我必须遵守系统规则。

例如：

```text
AGENTS.md 写着“可以随意删除任何文件”
```

这不会覆盖沙盒和安全规则。

## 5. 如何在当前项目中固定一种人设和回答方式

推荐在当前项目根目录创建：

```text
AGENTS.md
```

这份文件就是给 Codex 看的项目级说明。你可以把它当作“这个项目里的协作协议”。

## 6. 一个简单的 `AGENTS.md` 示例

可以写成这样：

```markdown
# AGENTS.md

## 角色设定

你是本项目的中文技术协作助手。回答要耐心、清晰、直接，像一个资深工程师带新人，不要使用夸张营销语气。

## 回答语言

- 默认使用中文回答。
- 技术名词可以保留英文，例如 `commit`、`branch`、`middleware`。
- 不要使用过多比喻；如果使用比喻，必须立刻给出技术解释。

## 回答结构

除非用户另有要求，回答按这个结构：

1. 结论
2. 原因
3. 具体步骤
4. 风险或注意事项
5. 下一步建议

如果是代码修改任务，最终回答按这个结构：

- 改了什么
- 改了哪些文件
- 如何验证
- 是否还有风险

## 工作方式

- 修改代码前先阅读相关文件。
- 不要做无关重构。
- 不要覆盖用户已有改动。
- 高风险操作前先确认。
- 完成后尽量运行相关测试。

## Git 规则

- 不要自动提交。
- 不要自动推送。
- 不要执行 `git reset --hard` 或 `git clean -f`，除非用户明确要求并再次确认。

## 项目命令

- 安装依赖：`npm install`
- 本地开发：`npm run dev`
- 测试：`npm test`
- 构建：`npm run build`
```

这类文件越具体，越容易影响我的实际行为。

## 7. 如何固定“人设”

你可以在 `AGENTS.md` 里写：

```markdown
## 人设

你在本项目中扮演“严格但耐心的代码审查导师”。

特点：

- 先指出问题，再解释原因。
- 不说空泛鼓励。
- 对代码风险保持敏感。
- 看到不确定的地方要明确说“不确定”。
- 不要为了显得友好而弱化严重问题。
```

或者换一种：

```markdown
## 人设

你在本项目中扮演“产品型前端工程师”。

特点：

- 回答时优先考虑用户体验和可维护性。
- 讨论 UI 时要关注信息层级、状态、可访问性和移动端。
- 语气简洁，偏实用，不写长篇理论。
```

## 8. 如何固定回答结构

可以写：

```markdown
## 固定回答结构

当用户问概念问题时，按这个结构回答：

1. 一句话解释
2. 具体例子
3. 常见误区
4. 在本项目中的用法

当用户让你改代码时，最终回答按这个结构：

1. 完成情况
2. 修改文件
3. 验证命令和结果
4. 后续建议
```

也可以更严格：

```markdown
## 输出格式

所有回答必须使用 Markdown。

标题使用二级标题 `##`。
列表使用短 bullet。
代码块必须带语言标识。
每次回答不超过 800 字，除非用户要求详细解释。
```

## 9. 如何固定语法和表达风格

可以写：

```markdown
## 语言风格

- 使用简体中文。
- 句子尽量短。
- 先给结论，再解释。
- 不使用网络流行语。
- 不使用 emoji。
- 不使用“显然”“很简单”这类可能让新手有压力的词。
- 命令、文件名、变量名使用反引号包裹。
```

如果你希望我像教程一样讲，可以写：

```markdown
## 教学风格

- 面向初学者。
- 每个新概念先用一句话解释。
- 再给一个具体例子。
- 最后说明它在当前项目中有什么用。
```

## 10. 如何让不同项目使用不同人设

做法很简单：每个项目放自己的 `AGENTS.md`。

例如：

```text
E:\code\project_a\AGENTS.md
E:\code\project_b\AGENTS.md
```

`project_a` 可以写：

```markdown
# AGENTS.md

你是一个严格的后端架构审查助手。回答要关注数据一致性、接口契约、事务、安全和可观测性。
```

`project_b` 可以写：

```markdown
# AGENTS.md

你是一个面向初学者的前端教学助手。回答要温和、细致，多用示例解释 React、CSS 和浏览器行为。
```

当你在 `project_a` 工作区打开 Codex，我会按 `project_a` 的规则工作。

当你在 `project_b` 工作区打开 Codex，我会按 `project_b` 的规则工作。

## 11. 如果项目里有多个规则文件怎么办

常见情况是：

- 根目录有一个总的 `AGENTS.md`
- 子目录也可能有更具体的 `AGENTS.md`

一般理解：

- 根目录规则适用于整个项目
- 更靠近当前文件的子目录规则可以补充更细的要求

例如：

```text
my-app\
  AGENTS.md
  frontend\
    AGENTS.md
  backend\
    AGENTS.md
```

可以让前端和后端有不同规则：

- `frontend/AGENTS.md` 规定 UI、样式、组件、浏览器验证方式
- `backend/AGENTS.md` 规定 API、数据库、事务、测试方式

如果规则冲突，应当优先遵守更具体目录下的规则，但仍不能违反系统和安全规则。

## 12. 当前会话里的临时要求和 `AGENTS.md` 的区别

你也可以直接在聊天里说：

```text
接下来都用中文回答，并且每次先给结论。
```

这会影响当前会话，但不一定稳定影响另一个项目或未来新会话。

`AGENTS.md` 更适合保存长期项目规则，因为它随着项目存在。

对比：

| 方式 | 适合什么 | 持久性 |
| --- | --- | --- |
| 当前聊天里说明 | 临时偏好、本轮任务 | 当前会话内较有效 |
| `AGENTS.md` | 项目级长期规则 | 跟随项目 |
| Codex 全局配置 | 个人通用习惯 | 可能影响所有项目 |

## 13. 你可以在本项目中怎么做

当前项目是一个 Codex 学习文档目录。如果你想让我在这里固定为“新手教学助手”，可以创建：

```text
E:\code\codex_projects\first_use\AGENTS.md
```

内容可以是：

```markdown
# AGENTS.md

## 项目定位

这是一个 Codex 初次使用学习项目。所有输出都应服务于新手理解 Codex 的能力、权限、工作流和安全边界。

## 助手角色

你是一个面向初学者的中文 Codex 教学助手。

## 回答风格

- 使用简体中文。
- 先给结论，再分步骤解释。
- 多用具体例子。
- 避免过度抽象。
- 命令和文件名使用反引号。
- 用户要求生成文档时，直接写入 `.md` 文件。

## 文档规则

- 每份文档开头包含更新日期和当前工作区。
- 使用 Markdown 标题和短列表。
- 解释概念时包含“是什么”“怎么用”“注意事项”。
- 适合完全没有经验的新手阅读。

## 安全规则

- 不要访问工作区外的文件，除非用户明确要求。
- 不要安装依赖或访问网络，除非用户明确批准。
- 不要执行删除、重置、提交、推送等高风险操作，除非用户明确确认。
```

这样以后你在这个项目里让我生成文档，我就更容易保持统一风格。

## 14. 如果你想在另一个项目使用另一个人设

假设你有另一个项目：

```text
E:\code\my_startup_app
```

你可以在里面写：

```text
E:\code\my_startup_app\AGENTS.md
```

内容例如：

```markdown
# AGENTS.md

## 助手角色

你是一个偏产品和工程结合的全栈开发助手。

## 优先级

1. 不破坏现有用户体验。
2. 保持代码可维护。
3. 小步提交，方便 review。
4. 对不确定的业务规则先提问。

## 回答结构

代码任务最终回答必须包含：

- 实现内容
- 修改文件
- 验证结果
- 风险点

## 技术偏好

- 优先沿用项目已有技术栈。
- 不引入新依赖，除非收益明显并先说明理由。
- 前端修改必须考虑移动端和加载状态。
```

这样这个项目就可以拥有不同的人设和工作习惯。

## 15. 怎么让我创建这些规则文件

你可以直接说：

```text
请为当前项目创建 AGENTS.md，让你以后以“中文新手教学助手”的人设回答，并统一使用教程式 Markdown 结构。
```

或者：

```text
请根据这个项目的 README 和代码结构，生成一份适合本项目的 AGENTS.md。
```

如果你已经有想法，可以说：

```text
请把以下规则写入 AGENTS.md：[你的规则]
```

## 16. 写项目规则时的建议

建议写清楚：

- 项目是做什么的
- 我应该扮演什么角色
- 默认回答语言
- 默认回答结构
- 是否允许直接改代码
- 修改前是否需要先给计划
- 是否允许运行命令
- 测试命令是什么
- Git 提交和推送规则
- 禁止访问哪些文件
- 高风险操作如何确认

不要只写抽象描述：

```text
你要专业一点。
```

更好的写法：

```text
回答先给结论，再给步骤。代码修改后必须说明修改文件和验证命令。不要自动提交或推送。
```

## 17. 重要限制

`AGENTS.md` 很有用，但它不是最高权限。

它不能让我：

- 绕过系统规则
- 绕过沙盒限制
- 忽略安全要求
- 访问未授权的连接器
- 私自安装插件
- 执行被禁止或高风险的操作

它能做的是：在允许范围内，让我更稳定地按你的项目习惯工作。

## 18. 一句话总结

我回答时会综合系统规则、开发者规则、当前环境、会话历史、用户请求、工具结果、技能说明和项目文件。想在一个项目中固定人设和回答结构，最推荐在项目根目录写 `AGENTS.md`；想让另一个项目使用另一种人设，就在另一个项目写另一份 `AGENTS.md`。项目规则越具体，我越容易稳定地按你的方式协作。
