## 1. 先说结论

在当前会话里，Codex 不是“随便控制整台电脑”的程序。它的能力由几个边界共同限制：

- 当前工作区
- 文件系统沙盒
- 命令执行权限
- 网络权限
- 已安装插件和连接器
- 你对高风险操作的批准

当前最重要的一条是：我可以读取本机文件，但默认只允许在当前工作区和明确可写目录内修改文件。当前可写根目录是：

```text
E:\code\codex_projects\first_use
```

也就是说，我可以在这个目录下创建、修改、删除文件；如果要写入其他位置，通常需要额外授权。

## 2. 我能访问哪些文件

### 2.1 当前工作区内的文件

当前工作区是：

```text
E:\code\codex_projects\first_use
```

我可以在这里：

- 列出文件和目录
- 读取文件内容
- 搜索文本
- 创建新文件
- 修改已有文件
- 删除文件，但涉及危险删除时会更谨慎，通常需要明确授权
- 运行项目命令，例如测试、构建、格式化、启动本地服务

本文件和上一份 `codex_intro.md` 就是写在这个工作区里的。

### 2.2 工作区外的文件

当前环境说明允许我读取文件，但写入受到沙盒限制。

这意味着：

- 我可能可以读取工作区外的普通文件
- 我默认不能随意修改工作区外文件
- 如果需要写入工作区外目录，需要请求你的批准
- 对系统目录、用户配置目录、其他项目目录，应当默认视为高风险区域

举例：

```text
C:\Users\你的用户名\Documents
C:\Users\你的用户名\.ssh
C:\Users\你的用户名\.env
C:\Windows
D:\其他项目
```

这些地方即使技术上可能可读，我也不应该无目的地扫描。你也不应该让我读取包含密钥、个人隐私或敏感业务数据的文件，除非你明确知道自己在做什么。

## 3. 我能对文件做哪些操作

### 3.1 读取

我可以读取你授权环境中可访问的文件内容，例如：

- 源代码
- README
- 配置文件
- 日志
- 测试结果
- Markdown、JSON、YAML、CSV 等文本文件

我通常会优先读取和任务直接相关的文件，而不是无差别扫描整个电脑。

### 3.2 搜索

我可以使用搜索命令在目录里查找内容，例如：

```bash
rg "keyword"
rg --files
```

这适合定位函数、配置、报错信息、路由、组件、测试文件等。

### 3.3 创建和修改

在当前工作区内，我可以创建和修改文件，例如：

- 新增代码文件
- 修改已有代码
- 生成文档
- 更新测试
- 修改配置
- 调整样式

我会尽量保持改动范围和你的任务相关，不主动做无关重构。

### 3.4 删除

删除属于高风险操作。即使在工作区内，我也应当谨慎处理。

通常只有在这些情况下才会删除文件：

- 你明确要求删除
- 文件是我刚刚生成的临时文件
- 删除是完成任务的必要步骤，并且风险明确可控

递归删除、大范围清理、清空目录、重置 Git 工作区等操作应当先经过你明确确认。

## 4. 我能运行哪些命令

我可以通过 PowerShell 在当前环境中运行命令。当前 shell 是：

```text
powershell
```

常见可运行命令包括：

- 查看目录：`Get-ChildItem`
- 读取文件：`Get-Content`
- 搜索文本：`rg`
- 查看 Git 状态：`git status`
- 运行测试：`npm test`、`pytest`、`cargo test` 等
- 运行构建：`npm run build` 等
- 启动本地开发服务器

但是，命令执行仍然受沙盒和审批系统限制。

## 5. 哪些操作需要你批准

当前规则下，以下操作通常需要额外批准或非常谨慎：

### 5.1 写入工作区外目录

例如修改：

```text
C:\Users\你的用户名\.gitconfig
C:\Users\你的用户名\.ssh\config
C:\Windows\System32
D:\其他项目
```

这类操作超出当前工作区，需要明确授权。

### 5.2 访问网络或下载依赖

当前网络访问是受限的。比如：

- `npm install`
- `pip install`
- `cargo fetch`
- `git pull`
- 下载远程资源
- 访问包管理器或外部 API

如果任务确实需要网络，我会请求你批准。

### 5.3 打开 GUI 程序

例如：

- 打开浏览器窗口
- 打开系统文件管理器
- 启动桌面应用

这类操作通常需要更高权限或使用 Codex app 的内置浏览器能力。

### 5.4 破坏性操作

高风险命令包括：

```bash
git reset --hard
git checkout -- .
Remove-Item -Recurse
rm -rf
格式化磁盘
清空目录
批量移动或覆盖文件
```

除非你非常明确地要求，否则我不应该执行这些操作。即使你要求了，我也应当先确认目标路径和影响范围。

### 5.5 涉及密钥和凭证的操作

包括：

- 读取 `.env`
- 读取 SSH 私钥
- 读取 API key
- 修改认证配置
- 创建或删除 token
- 登录第三方服务

这类操作应最小化，并且只在任务确实需要时进行。

## 6. 我不能做什么

在当前会话和权限模型下，我不能或不应该做这些事：

- 无限制控制你的整台电脑
- 绕过系统权限或沙盒
- 在没有授权的情况下修改工作区外文件
- 私自安装软件或依赖
- 私自上传你的文件
- 私自访问受保护的外部服务
- 保证所有生成代码 100% 正确
- 替你决定高风险业务、法律、财务或安全结论

如果某个操作被系统阻止，我不会绕过限制，而是会请求你批准或说明阻碍。

## 7. 插件和连接器会扩大哪些访问范围

Codex app 可以通过插件或连接器访问更多资源。当前环境中可见的能力包括：

- GitHub：查看仓库、PR、issue、评论、CI 状态，在授权范围内创建评论、PR 或合并等
- Browser：使用 Codex 内置浏览器访问和检查本地页面
- Automations：创建提醒、周期任务或后续跟进
- Image generation：生成或编辑图片资源
- 本地 shell：运行 PowerShell 命令

如果你连接了 Gmail、Google Drive、Slack、Notion、日历等插件，它们也可能让 Codex 读取或操作对应服务中的数据。但前提是：

- 连接器已安装并授权
- 当前任务需要使用它
- 工具暴露了相应能力

授权外部连接器时，你应当把它当成给一个自动化助手访问某项服务的权限，而不是普通聊天。

## 8. 你如何控制我的权限

### 8.1 控制工作区

最简单有效的办法：只在你希望 Codex 处理的项目目录里打开它。

建议：

- 为每个项目单独开工作区
- 不要把整个用户主目录作为工作区
- 不要把包含大量私人文件的目录作为工作区
- 对实验任务使用临时目录

当前工作区就是一个相对安全的示例：

```text
E:\code\codex_projects\first_use
```

### 8.2 使用 Git

建议在使用 Codex 前检查：

```bash
git status
```

如果项目还不是 Git 仓库，可以先初始化或备份。

Git 的好处是：

- 可以查看 Codex 改了什么
- 可以逐行 review
- 可以回滚不满意的改动
- 可以用分支隔离实验

### 8.3 明确告诉 Codex 不要做什么

你可以直接写约束：

```text
只读取当前目录，不要访问其他目录。
```

```text
不要修改文件，先只做分析。
```

```text
不要运行安装命令。
```

```text
不要读取 .env、密钥文件或用户目录。
```

```text
不要提交、推送或创建 PR，除非我明确要求。
```

这类约束越明确，我越容易按你的安全边界工作。

### 8.4 对高风险命令保持审批

当我请求批准时，你可以选择拒绝。

尤其要谨慎批准：

- 安装依赖
- 访问网络
- 删除文件
- 写入工作区外路径
- Git reset、clean、checkout
- 修改系统配置
- 读取或写入密钥文件

如果你不确定，可以让我先解释命令目的和影响范围。

### 8.5 使用只读模式思路

即使工具本身允许写入，你也可以通过提示词要求我只读：

```text
本轮只允许分析，不允许修改文件或运行写入命令。
```

适合场景：

- 初次阅读陌生项目
- 审查安全问题
- 评估重构方案
- 高风险生产代码
- 你还没有备份或提交当前改动

### 8.6 控制外部连接器授权

对于 GitHub、Slack、Google Drive、邮件、日历等连接器：

- 只授权真正需要的服务
- 定期检查授权列表
- 不需要时撤销授权
- 对企业数据使用组织批准的账号和策略
- 让 Codex 在操作前说明它要访问什么

## 9. 推荐的安全使用方式

### 9.1 普通代码任务

可以这样说：

```text
请只在当前项目目录内工作。可以读取和修改项目文件，可以运行测试。不要访问用户目录、不要安装依赖、不要提交代码。
```

### 9.2 只读分析任务

```text
请只读分析当前仓库，不要修改任何文件，不要运行会写入磁盘的命令。请总结项目结构和潜在问题。
```

### 9.3 高风险修改任务

```text
这个任务涉及核心业务逻辑。请先只分析并给出修改方案、影响范围、需要测试的点。等我确认后再改代码。
```

### 9.4 有敏感文件的项目

```text
不要读取 .env、*.pem、*.key、id_rsa、secrets、credentials 等文件。需要配置值时请告诉我变量名，不要读取具体值。
```

### 9.5 希望严格控制命令

```text
运行任何安装、删除、网络、Git 写入或工作区外写入命令前，都必须先询问我。
```

## 12. 一句话总结

当前 Codex 的权限不是无限的：它能在工作区里读写和运行命令，能在可访问范围内读取相关文件，但写入工作区外、联网、安装依赖、GUI 操作和破坏性命令通常需要你的批准。你控制权限的最好方式，是选对工作区、使用 Git、明确约束、谨慎批准高风险操作，并避免把密钥和隐私数据暴露给任务上下文。
