# Codex 初次使用指南

> 更新日期：2026-05-08  
> 适用对象：第一次接触 OpenAI Codex，想知道它是什么、怎么用、能做什么、有哪些限制的人。

## 1. Codex 是什么

Codex 是 OpenAI 面向软件开发场景的 AI 编程代理。它不是单纯的聊天机器人，而是可以围绕一个代码仓库完成实际工作的协作式开发助手：阅读代码、理解项目结构、修改文件、运行命令、执行测试、解释报错、做代码审查，甚至把较独立的任务交给云端后台处理。

你可以把 Codex 理解成一个“会读仓库、会动手、会和你确认风险”的结对工程师。它最擅长的不是替你凭空猜业务，而是在已有项目、明确目标和可验证反馈的条件下推进工作。

官方资料入口：

- [Using Codex with your ChatGPT plan](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan/)
- [OpenAI Codex CLI Getting Started](https://help.openai.com/en/articles/11096431-openai-codex-ci-getting-started)
- [Introducing Codex](https://openai.com/index/introducing-codex/)
- [OpenAI Codex GitHub repo](https://github.com/openai/codex)

## 2. Codex 可以在哪里使用

Codex 目前主要有几种使用形态：

| 使用方式 | 适合场景 |
| --- | --- |
| Codex CLI | 在终端里让 Codex 阅读、编辑、运行和验证本地项目 |
| IDE 扩展 | 在 VS Code、Cursor、Windsurf 等编辑器里边写代码边协作 |
| Codex app | 在桌面应用里管理多个项目、多个代理、工作区、技能、自动化和 Git 操作 |
| Codex web / 云端任务 | 把相对独立的任务委派给云端沙盒后台运行 |
| GitHub Code Review | 在 GitHub PR 中让 Codex 自动或手动进行代码审查 |
| Codex SDK / Slack 集成 | 用程序化方式或团队工具触发 Codex 工作流 |

如果你正在这个桌面环境里使用我，那你已经在使用 Codex app 形态：我可以看到当前工作区、运行终端命令、编辑文件、使用可用工具，并把结果落到本地文件里。

## 3. Codex 能做什么

Codex 的能力可以分成几类。

### 3.1 理解代码

它可以帮你：

- 解释一个陌生仓库的目录结构和启动方式
- 找到某个功能对应的文件、函数、组件或路由
- 梳理调用链、数据流、配置来源和依赖关系
- 解释一段复杂代码为什么这样写
- 对比两个实现方案的利弊

示例提示词：

```text
请阅读这个项目，告诉我它的入口文件、主要模块、启动方式和测试方式。
```

```text
帮我追踪用户登录流程，从前端表单到后端校验都经过哪些文件。
```

### 3.2 修改代码

它可以帮你：

- 新增功能
- 修复 bug
- 重构重复逻辑
- 补充类型、边界处理和错误提示
- 修改 UI、文案、样式和交互
- 迁移 API、依赖或配置

示例提示词：

```text
请实现“导出 CSV”功能，保持现有 UI 风格，并补充必要测试。
```

```text
这个页面在移动端布局错位，请定位原因并修复。
```

### 3.3 运行和验证

Codex 可以在允许范围内运行命令，例如：

- `npm test`
- `pnpm lint`
- `pytest`
- `cargo test`
- `go test ./...`
- 本地开发服务器
- 构建命令

它通常会先查看项目脚本、README、配置文件或 `AGENTS.md`，再选择合适的验证方式。

### 3.4 代码审查

Codex 可以以 code review 的姿态工作，重点关注：

- 行为回归
- 安全风险
- 并发或边界问题
- 缺失测试
- 类型和契约不一致
- 性能和可维护性风险

示例提示词：

```text
请 review 当前改动，只列出真实风险和需要修改的问题。
```

### 3.5 Git 和协作

在具备相应权限时，Codex 可以协助：

- 查看 `git status`、diff、commit 历史
- 总结本次改动
- 写 commit message
- 创建 PR 描述
- 处理 PR review comments
- 调试 GitHub Actions 失败日志

是否真正提交、推送、合并，通常应由你明确授权。

### 3.6 多模态和工具能力

根据当前环境和权限，Codex 还可能支持：

- 读取截图或图片来修 UI
- 使用浏览器检查本地页面
- 生成或编辑图片资产
- 设置自动化提醒或周期任务
- 连接 GitHub、Slack、日历、邮件、文档等外部工具

具体可用能力取决于你安装的客户端、插件、连接器、权限和订阅计划。

## 4. 怎么开始使用

### 4.1 本地项目中使用

推荐流程：

1. 打开项目根目录。
2. 确认项目在 Git 管理下，方便查看和回滚改动。
3. 启动 Codex CLI、IDE 扩展或 Codex app。
4. 先让 Codex 阅读项目，不要一上来就让它大改。
5. 给出清晰任务和验收标准。
6. 让 Codex 修改、运行测试、总结结果。
7. 你 review diff，再决定是否提交。

示例：

```text
先熟悉这个仓库。请告诉我项目类型、启动命令、测试命令，以及你会优先阅读哪些文件。
```

```text
请修复订单列表分页失效的问题。验收标准：切换页码时 URL 和列表数据都更新，现有测试通过。
```

### 4.2 Codex CLI 的常见启动方式

官方 Help Center 提到 Codex CLI 可以通过 npm 安装和升级：

```bash
npm install -g @openai/codex
codex
codex --upgrade
```

早期 CLI 文档中还提到过几种审批模式：

| 模式 | 大致含义 |
| --- | --- |
| Suggest | 主要读取和建议，修改或命令需要你确认 |
| Auto Edit | 可以自动编辑文件，但运行命令仍需确认 |
| Full Auto | 在受限沙盒中更自主地读写和运行命令 |

不同版本的 CLI、IDE 扩展和 Codex app 可能会调整名称、默认模型和交互方式。实际使用时以你当前客户端显示为准。

### 4.3 在 Codex app 里使用

如果你在桌面 Codex app 中，可以这样用：

- 把项目作为工作区打开
- 直接描述目标，让 Codex 阅读、编辑、运行命令
- 对前端项目，让 Codex 启动本地服务并用内置浏览器检查页面
- 对较大任务，让 Codex 拆成步骤推进
- 对重复任务，可以使用 Automations
- 对长期偏好或项目约定，可以写入 README、`AGENTS.md` 或项目文档

## 5. 写好提示词的关键

Codex 最喜欢的信息是：目标、上下文、约束、验收标准。

不太好的写法：

```text
帮我优化一下。
```

更好的写法：

```text
请优化用户列表页面的首屏加载速度。不要改变现有视觉样式。优先检查接口请求、重复渲染和分页逻辑。完成后运行现有前端测试，并说明你改了哪些文件。
```

建议你在任务里说明：

- 你想解决什么问题
- 哪些行为不能改变
- 是否允许重构
- 是否需要测试
- 是否需要启动本地服务验证
- 你希望它先解释、先计划，还是直接实现

## 6. Codex 的限制

Codex 很强，但不是无限制的。初次使用尤其要记住这些边界。

### 6.1 它可能会误解业务意图

Codex 能读代码，但不自动知道你的真实产品规则。业务规则、权限边界、财务口径、法律要求、历史兼容性，最好明确写出来。

### 6.2 它依赖可见上下文

如果关键代码、文档、环境变量、私有服务或数据库不可见，它只能基于当前能访问的内容推断。上下文越完整，结果越可靠。

### 6.3 它运行命令受权限和沙盒限制

本地环境通常有文件读写、网络访问、命令执行等限制。某些操作需要你批准，例如：

- 安装依赖
- 访问网络
- 打开 GUI 程序
- 写入工作区外的目录
- 执行潜在破坏性命令

这是好事：它让 Codex 的自动化能力更可控。

### 6.4 它不能保证一次写对

和真人工程师一样，Codex 也可能：

- 漏掉边界情况
- 写出不符合项目风格的代码
- 修复一个问题但引入另一个问题
- 测试不足
- 对过时依赖或冷门框架判断不准

因此重要改动一定要 review diff、跑测试、做必要的人工验收。

### 6.5 使用量和模型能力受计划影响

Codex 的可用额度和限制取决于你的 ChatGPT 计划、任务规模、执行位置以及当前产品政策。小脚本和单文件修改消耗较少，大型仓库、长会话、多文件任务和云端任务会消耗更多。

官方 Help Center 当前说明 Codex 可随 ChatGPT Plus、Pro、Business、Enterprise/Edu 等计划使用，并且部分时期可能有临时访问或额度活动。由于这些政策会变化，涉及价格、额度、可用模型时应查看官方页面。

### 6.6 数据和隐私要主动管理

Codex 可能处理你的代码、提示词、终端输出、截图、连接器数据或云端任务上下文。不同计划的数据使用政策不同，企业和个人账户的数据控制也不同。

建议：

- 不要把不必要的密钥、令牌、客户隐私数据贴给 Codex
- 使用 `.env`、密钥管理器或平台密钥机制，不要硬编码 secrets
- 对敏感仓库，先确认组织的数据控制、保留策略和连接器权限
- 需要云端执行时，确认仓库和环境是否适合被委派到云端沙盒

## 7. 第一次使用的建议

### 7.1 从小任务开始

第一次不要直接让 Codex 重构整个系统。可以从这些任务开始：

- 解释项目结构
- 修一个明确 bug
- 补一个单元测试
- 改一处 UI 文案
- 为某个函数加类型或边界处理
- 生成 README 的使用说明

### 7.2 先让它读，再让它改

一个很稳的开场是：

```text
请先阅读项目，不要修改文件。告诉我你理解的项目结构、启动方式、测试方式，以及完成这个任务可能要改哪些地方。
```

等它说明合理后，再说：

```text
按这个方向实现。改动保持最小，完成后运行相关测试。
```

### 7.3 给出验收标准

比如：

```text
验收标准：
1. 用户提交空邮箱时显示错误提示。
2. 合法邮箱可以正常提交。
3. 不改变现有表单布局。
4. 相关测试通过。
```

验收标准越清楚，Codex 越容易收敛。

### 7.4 用 Git 保护自己

建议在干净工作区中开始：

```bash
git status
```

如果工作区已有你的未提交改动，告诉 Codex：

```text
当前有我未提交的改动。请不要覆盖或回退它们，只在必要文件上做最小修改。
```

### 7.5 要求它验证

可以直接说：

```text
完成后请运行项目中最相关的测试或检查命令。如果不能运行，请说明原因。
```

### 7.6 对高风险任务分阶段

高风险任务包括：

- 数据库迁移
- 权限系统修改
- 支付、账务、订单逻辑
- 大规模重构
- 生产部署脚本
- 安全相关代码

这类任务建议分成：

1. 只分析，不改代码
2. 提出方案和风险
3. 小范围实现
4. 跑测试
5. 人工 review
6. 再扩大范围

## 8. 常用提示词模板

### 8.1 熟悉仓库

```text
请阅读这个仓库并总结：
1. 项目用途
2. 技术栈
3. 主要目录
4. 启动命令
5. 测试和 lint 命令
6. 新人最应该先看的文件
先不要修改文件。
```

### 8.2 修 bug

```text
请修复这个 bug：[描述问题]。
复现方式：[步骤]。
期望行为：[描述]。
约束：尽量最小改动，不改变无关逻辑。
完成后运行相关测试，并说明改动点。
```

### 8.3 加功能

```text
请实现：[功能描述]。
用户流程：[说明]。
验收标准：[列出标准]。
约束：[设计风格、兼容性、性能、安全要求]。
请先定位相关文件，再实现并验证。
```

### 8.4 做 review

```text
请 review 当前 Git 改动。重点找 bug、回归风险、安全问题和缺失测试。
不要泛泛评价风格，只列出需要处理的具体问题。
```

### 8.5 解释代码

```text
请解释这个模块的执行流程，重点说明数据从哪里来、在哪里变形、错误如何处理、对外暴露什么接口。
```

## 9. 适合交给 Codex 的任务

很适合：

- 明确 bug 修复
- 单元测试补全
- 类型补全
- 文档生成
- 小到中等范围的功能
- 前端样式和交互修复
- 根据报错定位问题
- 梳理陌生代码
- PR review

谨慎交给 Codex：

- 模糊的产品设计
- 大规模架构重写
- 无测试覆盖的核心交易链路
- 涉及合规、法律、财务结论的判断
- 需要访问真实生产数据的任务
- 无法在本地验证的复杂变更

## 10. 和 Codex 协作的最佳心态

把 Codex 当成一个很快、很勤奋、会犯错但愿意验证的工程搭档。

你负责：

- 定义目标
- 提供业务判断
- 控制风险
- review 关键改动
- 决定是否合并和发布

Codex 负责：

- 快速读代码
- 找相关文件
- 写初版实现
- 跑检查
- 总结改动
- 帮你从琐碎工作里省出注意力

最好的使用方式不是“让它替你做所有决定”，而是让它把探索、改动和验证的循环变快。你给方向，它给推进力；你把关，它出手。

## 12. 一句话总结

Codex 是一个面向真实软件工程流程的 AI 编程代理。它能读代码、改代码、跑命令、做 review、处理 GitHub 工作流，也能在某些环境中进行云端任务和自动化。它的价值来自“可执行的协作”，限制也来自“必须在上下文、权限、测试和人工把关内工作”。第一次使用时，从小任务开始，写清楚目标和验收标准，让它先理解，再动手。
