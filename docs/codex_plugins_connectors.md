> 更新日期：2026-05-08

## 0. 最简解释：用一个例子理解

如果你只记一个版本，可以先记这个：

```text
技能 = 我该怎么做事的说明书
插件 = 给我增加一组能力的工具包
连接器 = 你授权我访问某个外部服务的账号通道
工具 = 我实际按下去执行动作的按钮
```

举个例子，你说：

```text
帮我看看 GitHub 上这个 PR 为什么 CI 失败，并修一下。
```

这时可能会发生：

1. 我判断这是一个 GitHub PR + CI 调试任务。
2. 我启用 `github:gh-fix-ci` 这个技能，因为它告诉我处理 CI 失败时应该先看 PR、再看 Actions 日志、再本地复现。
3. 我使用 GitHub 插件提供的能力，因为它让我可以查看 PR、diff、评论和 CI 状态。
4. 如果你的 GitHub 已授权，GitHub 连接器让我能访问你有权限访问的仓库数据。
5. 我调用具体工具，比如“获取 PR 信息”“读取 workflow 日志”“运行本地测试”“编辑文件”。

所以这四个东西不是并列的“技能列表”：

- 技能负责流程和规则。
- 插件负责提供一类能力。
- 连接器负责外部服务授权。
- 工具负责真正执行某个动作。

再换一个生活化比喻：

```text
你要我做菜。

技能 = 菜谱，告诉我步骤。
插件 = 厨房设备包，比如烤箱、锅、搅拌机。
连接器 = 你给我的冰箱钥匙或超市账号授权。
工具 = 刀、锅铲、开火按钮这些具体动作。
```

没有技能，我也许能做，但流程可能不够专业。没有插件，我可能没有对应设备。没有连接器，我就不能访问外部服务里的材料。没有工具，我就只能解释，不能实际动手。

## 0.1 我会不会自动用它们

一般规则是：

- 任务明显需要时，我会主动使用已有技能或插件。
- 你明确指定某个插件、连接器或技能时，我会优先按你的要求来。
- 涉及外部账号、写入外部服务、安装插件、读取敏感数据或破坏性操作时，不应该静默执行，需要说明或请求确认。
- 如果现有能力不够，我会先说明缺口；如果有可安装的对应能力，并且你明确需要它，我可以请求你安装或授权。

例子：

```text
打开 localhost:3000 看看页面布局。
```

我会主动使用 Browser 插件。

```text
只根据本地文件分析，不要访问 GitHub。
```

我就不应该使用 GitHub 连接器。

```text
用 Google Drive 找项目文档。
```

如果当前没有 Google Drive 连接器，我会告诉你需要安装或授权，而不是自己偷偷访问。

## 1. 先说结论

插件、连接器和技能不是同一个东西，但它们都会扩展 Codex 的能力。

简单理解：

| 名称 | 是什么 | 作用 |
| --- | --- | --- |
| 技能 Skill | 一组本地说明书或工作流规则 | 告诉 Codex 遇到某类任务时应该怎么做 |
| 插件 Plugin | 一包本地能力，可能包含技能、工具、MCP 服务或应用能力 | 给 Codex 增加一类可调用能力 |
| 连接器 Connector / App | 授权 Codex 访问某个外部服务 | 让 Codex 能读写 GitHub、Slack、Google Drive、邮箱、日历等数据 |
| 工具 Tool | Codex 实际调用的函数或能力接口 | 例如运行终端命令、改文件、查 GitHub PR、打开浏览器 |

一句话：

```text
技能更像“操作手册”，插件更像“能力包”，连接器更像“外部账号授权”，工具是我真正执行动作的接口。
```

## 2. 技能是什么

技能是写给 Codex 的专门工作说明。它通常不是直接访问外部服务的权限，而是告诉我：

- 什么时候应该使用这套流程
- 应该先读哪些文档
- 应该调用哪些工具
- 应该遵守哪些安全规则
- 遇到失败时如何降级

比如当前会话里可以看到这些技能：

- `openai-docs`：当你问 OpenAI 产品、API、模型、Codex 文档时，用官方文档口径回答
- `imagegen`：当你需要生成或编辑位图图片时使用
- `browser-use:browser`：当你要求打开、检查、测试本地网页时使用内置浏览器
- `github:github`：当你需要处理 GitHub 仓库、PR、issue 时使用
- `github:gh-fix-ci`：当你要修 GitHub Actions CI 失败时使用
- `github:gh-address-comments`：当你要处理 PR review comments 时使用
- `github:yeet`：当你要把本地改动提交、推送并开 PR 时使用

技能不会神奇地扩大系统权限。它只是让我按照更专业、更稳定的流程工作。

## 3. 插件是什么

插件是一组能力的打包形式。一个插件可能包含：

- 一个或多个技能
- 可调用工具
- MCP server
- 外部应用入口
- 浏览器能力
- GitHub 能力
- 项目脚手架或模板

当前环境里可见的插件包括：

| 插件 | 当前作用 |
| --- | --- |
| Browser | 使用 Codex app 内置浏览器打开、检查、测试本地页面 |
| GitHub | 通过 GitHub 连接能力查看仓库、PR、issue、CI、评论等 |

插件像给 Codex 装上的工具箱。是否能使用某个工具箱，取决于它是否已经安装、是否在当前会话暴露了工具，以及是否有必要的授权。

## 4. 连接器是什么

连接器通常是对外部服务的授权入口。它让 Codex 可以在你授权范围内访问某个服务的数据或执行操作。

常见连接器可能包括：

- GitHub
- Slack
- Gmail
- Outlook
- Google Drive
- Google Calendar
- Notion
- Linear
- Teams
- SharePoint

例如 GitHub 连接器可能允许我：

- 查看仓库
- 读取 PR diff
- 获取 issue 评论
- 查看 CI 结果
- 评论 PR
- 创建 PR
- 在有权限时请求 review、加 label 或合并

连接器比普通技能更敏感，因为它可能涉及你的外部账号、组织数据、私有仓库、邮件、文档或团队沟通记录。

## 5. 技能、插件、连接器之间的关系

可以这样理解：

```text
你提出任务
  ↓
Codex 判断需要哪类能力
  ↓
如果有对应技能，就按技能说明工作
  ↓
如果需要实际操作，就调用工具
  ↓
如果工具要访问外部服务，就依赖插件或连接器授权
```

例子：

```text
请帮我修复这个 PR 的 CI 失败。
```

可能发生的流程：

1. 我识别这是 GitHub CI 调试任务。
2. 使用 `github:gh-fix-ci` 技能。
3. 调用 GitHub 工具查看 PR 信息和 CI 状态。
4. 必要时运行本地命令复现问题。
5. 修改代码。
6. 总结修复和验证结果。

这里：

- `github:gh-fix-ci` 是技能
- GitHub 插件提供相关能力
- GitHub 连接器提供仓库访问授权
- 具体 API 调用和 shell 命令是工具

## 6. 我会自己调用，还是必须你明确指定

分情况。

### 6.1 当任务明显匹配时，我会主动使用

如果你的请求明显属于某个技能或插件的范围，我会主动使用，而不需要你显式点名。

例如：

```text
请打开 localhost:3000 看一下页面有没有布局问题。
```

我会倾向使用 Browser 插件。

```text
请帮我 review 这个 GitHub PR。
```

我会倾向使用 GitHub 插件和对应 GitHub 技能。

```text
请介绍 OpenAI Responses API 的最新用法。
```

我会倾向使用 `openai-docs` 技能，并优先查官方资料。

### 6.2 当你明确指定时，我必须优先考虑

如果你明确说：

```text
使用 GitHub 插件查看这个 PR。
```

或者：

```text
用 browser 打开本地页面。
```

那我会优先按你的指定来做。除非当前插件不可用、权限不足或任务不适合，我会说明原因并选择替代方案。

### 6.3 当任务不需要插件时，我不会硬用

比如你只是让我生成一个本地 Markdown 文件，不需要 GitHub 或浏览器插件。我会直接使用本地文件编辑能力，不会为了“有插件”而调用插件。

### 6.4 当任务需要敏感权限时，我不会默默越权

如果任务涉及外部账号、私有仓库、邮件、密钥、删除文件、联网安装依赖等敏感操作，我应当说明并请求必要授权。

## 7. 如果当前插件或连接器不能满足要求，我会主动查找吗

### 7.1 我可以判断已有能力是否够用

我会先看当前可用工具、插件、技能和连接器能不能完成任务。

如果能完成，我会直接做。

如果不能完成，我会说明缺什么能力，例如：

```text

```

### 7.2 对已知可安装插件，我可以请求你安装

如果你明确要求使用某个插件或连接器，而它当前没有安装，但它在系统已知可安装列表中，我可以发起安装请求。

当前已知可请求安装的插件或连接器包括：

- Canva
- Chrome
- Figma
- Gmail
- Google Calendar
- Google Drive
- Linear
- Notion
- OpenAI Developers

- 当前环境没有对应能力
- 它确实在可安装列表里
- 任务需要它才能完成

### 7.3 我不会私自安装

我不能自己静默安装插件或连接器。安装和授权外部服务必须经过你。

尤其是 Gmail、Google Drive、Slack、Notion、Teams 这类连接器，涉及个人或组织数据，必须由你明确授权。

这种情况下我会：

- 说明当前没有对应插件
- 用已有工具完成可完成的部分
- 告诉你需要什么类型的连接器或授权
- 如有已知安装入口，再请求安装

## 9. 插件和连接器的安全边界

使用插件和连接器时要注意：

- 连接器可能访问你的私人或组织数据
- 不同连接器的读写权限不同
- 有些操作会产生外部可见影响，比如评论 PR、发送邮件、改日历、发 Slack 消息
- Codex 应该在执行外部写操作前让你知道
- 你可以随时拒绝安装、授权或执行

建议默认规则：

```text
读取外部服务前先说明要访问什么。任何会写入外部服务、发送消息、创建评论、修改文件、改日历、提交 PR 或合并代码的操作，都必须先得到我的明确确认。
```

## 11. 常见问答

### 11.1 插件就是技能列表吗

不是。

技能是“怎么做”的说明，插件是“能做什么”的能力包。一个插件可以包含多个技能，也可能暴露工具或连接器能力。

### 11.2 我不说插件名，你会不会用

会。只要任务明显需要某个插件或技能，我会主动使用。

例如你说“打开本地页面检查布局”，我会用 Browser；你说“看这个 GitHub PR”，我会用 GitHub。

### 11.3 你会不会乱用插件访问我的数据

不应该。正确行为是只在任务需要时使用相关插件，并且对外部服务写操作、高风险访问或敏感数据保持谨慎。

你可以明确限制：

```text
除非我明确说，否则不要使用外部连接器。
```

### 11.4 当前能力不够时你会怎么办

我会先用已有能力完成能完成的部分。如果确实缺少某个连接器或插件，并且它在可安装列表中，我可以请求你安装或授权。如果不可安装，我会说明限制和替代方案。

### 11.5 安装插件后你就能访问所有内容吗

不一定。还取决于：

- 你授权了哪个账号
- 连接器申请了哪些权限
- 当前工具暴露了哪些操作
- 外部服务本身的权限控制
- 当前任务是否需要使用它

## 12. 推荐你的默认控制方式

如果你希望插件和连接器使用更可控，可以对我说：

```text
默认优先使用本地工作区。只有当任务明确需要时才使用插件或连接器。读取外部服务前先说明要访问什么；任何外部写操作、发送消息、评论、创建文件、改日历、提交 PR 或合并代码前都必须先确认。
```

如果你希望更严格：

```text
本轮不要使用任何外部插件或连接器，只使用当前工作区文件和本地命令。
```

如果你希望更自动化：

```text
可以主动使用当前可用插件来完成任务，但所有写入外部服务的操作前都要确认。
```

## 13. 一句话总结

技能、插件和连接器是 Codex 扩展能力的三层机制：技能提供工作方法，插件提供能力包，连接器提供外部服务授权。我会在任务明显需要时主动使用已有技能或插件；你明确指定时我会优先按指定执行；如果当前能力不足，我会说明缺口，并在允许的情况下请求你安装或授权合适的插件或连接器，但不会私自安装或静默访问敏感服务。
