1

先认识 Codex 是什么

Codex 是会读项目、改文件、跑命令和解释结果的 AI 编程代理，不只是聊天助手。

它围绕项目工作

Codex 最擅长结合代码仓库、文件、命令输出和你的目标推进任务。

它可以动手

在权限允许时，它能创建文件、修改代码、运行测试和总结改动。

它仍需要你把关

业务规则、风险取舍和最终是否合并发布，应由你确认。

Codex 适合做哪些事
  • 解释陌生项目结构、启动方式和关键模块。
  • 修复明确 bug、补测试、改文档、做小到中等规模功能。
  • 根据报错定位问题，运行相关检查并解释结果。
  • 以 code review 视角找回归、安全、边界和测试缺口。
哪些事不要一开始就交给它
  • 完全模糊的产品方向，例如“帮我优化一下”。
  • 无测试覆盖的核心账务、支付、权限和生产部署逻辑。
  • 需要读取密钥、客户隐私或真实生产数据的任务。
新手关键心法：把 Codex 当成“能执行的工程搭档”，而不是“自动替你做所有决定的人”。
2

第一次使用：先读，再改，再验证

不要一上来就让 Codex 大改。先让它读项目，确认理解，再给小任务和验收标准。

先只读分析

让 Codex 说明项目结构、启动方式、测试方式和可能要改的文件。

任务要可验收

写清楚目标、不能改变什么、完成后怎么判断成功。

完成后要验证

让 Codex 运行相关测试或说明为什么不能运行。

推荐第一次任务流程
  1. 让 Codex 只读项目，不修改文件。
  2. 确认它对项目结构和命令的理解。
  3. 给一个小任务，例如改文档、修一个明确 bug、补一个测试。
  4. 要求它说明修改文件和验证方式。
  5. 你查看 diff，再决定是否保留。
好提示词长什么样

好的提示词包含目标、上下文、约束和验收标准。

请修复订单列表分页失效的问题。
复现方式：点击第 2 页后列表没有更新。
约束：不要改变现有视觉样式。
验收标准：页码、URL 和列表数据都正确更新，相关测试通过。
3

理解权限：知道 Codex 能碰哪里

当前会话的核心边界是工作区和沙盒。能读不等于应该读，能写也不等于随便写。

工作区是主范围

当前可写根目录是 E:\code\codex_projects\first_use。

敏感操作要确认

联网、安装依赖、写工作区外、删除文件和危险 Git 命令都要谨慎。

密钥不要暴露

不要让 Codex 无目的读取 .env、私钥、token 和客户隐私数据。

哪些操作需要特别小心
  • 写入工作区外目录，例如用户目录、系统目录、其他项目。
  • 联网下载或安装依赖，例如 npm install、pip install。
  • 删除、递归清理、批量移动、覆盖文件。
  • git reset --hard、git clean -f、git checkout -- .。
推荐安全约束 
默认只在当前工作区内工作。
不要读取密钥、隐私文件或用户目录。
不要安装依赖、访问网络、删除文件、提交或推送代码，除非我明确批准。
复杂任务先给方案，再执行。
最重要的安全规则：高风险操作前让 Codex 解释“要做什么、影响哪些路径、如何撤销”。
4

分清技能、插件、连接器

技能是流程说明，插件是能力包，连接器是外部账号授权，工具是真正执行动作的接口。

技能告诉我怎么做

例如处理 GitHub CI 失败时，先看 PR、日志，再本地复现。

插件提供能力包

例如 Browser 用来检查本地页面，GitHub 用来处理 PR 和 CI。

连接器代表授权

访问 GitHub、Slack、Drive 等外部服务，通常需要你授权。

用一个 GitHub PR 例子串起来
  1. 你说：帮我看这个 PR 为什么 CI 失败。
  2. Codex 识别任务，使用 github:gh-fix-ci 技能。
  3. GitHub 插件提供读取 PR、diff、CI 状态的能力。
  4. GitHub 连接器决定是否能访问你的仓库数据。
  5. 具体工具执行“获取 PR 信息”“读取日志”“编辑文件”等动作。
我会自动用它们吗
  • 任务明显需要时，会主动使用已有技能或插件。
  • 你明确指定某个插件或连接器时，会优先按你的要求。
  • 涉及外部写操作、安装、授权或敏感数据时，不应该静默执行。
  • 当前能力不够时，会说明缺口，并在合适时请求安装或授权。
5

用 Git 管理每一次修改

git status 是开始前后的安全检查，git diff 是你 review Codex 改动的主工具。

开始前看状态

git status 告诉你当前分支、已修改文件和未跟踪文件。

用分支隔离任务

建议每个 Codex 任务一个分支，例如 codex/fix-login。

提交前看差异

git diff 帮你确认是否只改了相关文件。

推荐命令顺序 
git status
git switch -c codex/task-name
# 让 Codex 修改
git status
git diff
# 运行测试
git add .
git commit -m "Describe the change"
不满意修改时怎么办
  • 丢掉某个文件未提交改动：git restore path/to/file。
  • 预览会删除哪些未跟踪文件：git clean -n。
  • git clean -f 和 git reset --hard 会丢内容，新手不要随手执行。
6

用 AGENTS.md 固定项目习惯

如果你想让 Codex 在不同项目里使用不同人设、结构和规则，把项目约定写进根目录的 AGENTS.md。

项目规则跟着项目走

当前项目一份 AGENTS.md，另一个项目可以有另一份。

写具体比写抽象有用

不要只写“专业一点”，要写回答结构、测试命令和禁止事项。

不能覆盖安全规则

AGENTS.md 不能让 Codex 绕过沙盒、权限和系统规则。

AGENTS.md 可以写什么
  • 助手角色：新手教学助手、严格 code reviewer、产品型前端工程师。
  • 回答结构：结论、原因、步骤、风险、下一步。
  • 项目命令：启动、测试、构建、格式化。
  • 安全规则：不读密钥、不自动提交、不执行危险命令。
一个简短模板 
# AGENTS.md

## 助手角色
你是面向初学者的中文 Codex 教学助手。

## 回答风格
- 先给结论，再分步骤解释。
- 多用具体例子。
- 命令和文件名使用反引号。

## 安全规则
- 不要访问工作区外文件，除非用户明确要求。
- 不要安装依赖、删除文件、提交或推送，除非用户明确确认。

术语词典

这些名称第一次看容易混。先理解大概含义，再在真实任务里慢慢熟悉。

workspace

工作区

Codex 当前围绕工作的项目目录。当前工作区是 E:\code\codex_projects\first_use。

sandbox

沙盒

限制 Codex 能读写和运行什么的安全边界。当前模式允许在工作区内写文件。

skill

技能

一份工作流程说明。它告诉 Codex 遇到某类任务时按什么步骤做。

plugin

插件

一包能力，例如 Browser 或 GitHub。插件可能包含技能、工具或外部服务能力。

connector

连接器

你授权 Codex 访问外部服务的通道，例如 GitHub、Slack、Google Drive。

tool

工具

Codex 实际调用的动作接口，例如运行命令、编辑文件、查看 PR。

AGENTS.md

项目规则文件

写在项目根目录，告诉 Codex 本项目的人设、回答结构、命令和安全约束。

git status

Git 状态检查

查看当前分支、修改文件、未跟踪文件和暂存区状态，不会改动任何文件。

常用提示词

这些提示词可以直接复制给 Codex，用来控制范围、保护文件和提高任务质量。

首次阅读项目

请先只读分析当前仓库，不要修改文件。告诉我项目用途、技术栈、主要目录、启动命令、测试命令，以及新人最应该先看的文件。

修复明确 bug

请修复这个 bug：[描述问题]。复现方式：[步骤]。期望行为：[描述]。约束：尽量最小改动，不改变无关逻辑。完成后运行相关测试，并说明改动点。

限制访问范围

默认只在当前工作区内工作。不要读取密钥、隐私文件或用户目录。不要写入工作区外路径。不要安装依赖、访问网络、删除文件、提交或推送代码，除非我明确批准。

检查 Git 状态

请先运行 git status，告诉我当前工作区是否干净。不要修改文件。如果有未提交改动，请说明哪些可能是我自己的改动。

让 Codex 做 review

请 review 当前 Git 改动。重点找 bug、回归风险、安全问题和缺失测试。不要泛泛评价风格，只列出需要处理的具体问题。

创建项目规则

请为当前项目创建 AGENTS.md，让你以后以“中文新手教学助手”的人设回答，并统一使用教程式 Markdown 结构。安全规则包括：不读密钥、不自动提交、不执行危险命令。

安全提醒

使用 Codex 的关键不是把权限全放开，而是知道什么时候该让它动手，什么时候该先确认。

不要默认读取敏感文件

.env、私钥、token、客户数据、生产配置都应最小化暴露。

不要让它静默做外部写操作

评论 PR、发消息、改日历、写 Drive 文档、提交推送前都应明确确认。

不要随手执行危险 Git 命令

git reset --hard 和 git clean -f 会丢内容，执行前先看影响范围。

优先让它小步修改

每次只做一个明确任务，改完看 git diff，通过测试后再提交。

优先写验收标准

告诉 Codex 什么算成功，什么行为不能变，比单纯说“优化一下”可靠得多。

高风险任务先要方案

数据库、权限、支付、部署和大重构，先分析方案，再允许修改。
